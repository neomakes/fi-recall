# FI:RECALL 교리 온톨로지 (핵심 IP)

> 지저분한 자연어(SOP·복기)를 **하나의 안정된 교리 그래프**로 묶는 계약.
> 두 기능 — ① 교리 지도화 / ② 복기 반영 — 이 공유하는 단일 IP이며, 제품의 방어선이다.

- `doctrine-graph.schema.json` — 형식 계약 (JSON Schema 2020-12). Claude 구조화 출력의 강제 스키마.
- `example.sop-107.json` — 실 데이터 워크드 예시 (스키마 검증 + 추출 few-shot 겸용).
- 출처: `input/재난현장표준작전절차.pdf` (소방청, 275p, 5개 분야 + SSG).

---

## 왜 이게 IP인가

평범한 LLM 래퍼는 "SOP 넣으면 요약" 수준에 머문다. FI:RECALL의 해자는 **두 갈래의 지저분한 자연어가 같은 좌표계 위에서 만나게 하는 온톨로지**에 있다:

```
        ① SOP 원문  ─추출→ ┐
                          ├─→  [교리 그래프 = 공유 좌표계]  ─→  ② 수정안 / drill
        ② 현장 복기 ─grounding→ ┘
```

이 좌표계가 없으면 복기는 "어디에 닿는지" 말할 수 없고, 재발도 "같은 지점인지" 판정할 수 없다. 온톨로지의 4가지 설계 결정이 이를 가능케 한다.

### 설계 결정 1 — 출처 고정 결정론적 ID (`SOP-107-2.6`)
원문은 모든 조항에 **십진 계층 번호**(`1 → 1.2 → 1.2.3`)를 이미 부여하고 있다. 노드 id를 LLM이 짓게 하지 않고 **이 번호에서 결정론적으로 생성**한다. 효과:
- grounding(②)이 **재현 가능·감사 가능** — 같은 복기는 항상 같은 노드 id로 떨어진다.
- 재발 추적이 **node_id 단위 join**으로 환원된다(확률적 매칭에 의존하지 않음).
- 환각이 곧 인명 위험인 도메인에서 **모든 노드가 원문 페이지·인용으로 역추적**된다(`provenance`).

### 설계 결정 2 — Deontic Modality를 1급 속성으로
같은 위반이라도 조항의 **강제력**에 따라 고리를 닫는 방향이 갈린다. 그래서 `modality`(의무/권고/재량/정보)를 노드마다 추출한다. 이것이 **격차 분류의 경첩**이다 (설계 결정 4 참조).

> 실증: SOP-107-2.6은 다수 단위대 채널 분리를 *"할 수 있다"*(재량)로만 둔다. input/01의 무전 채널 혼선은 현장이 SOP를 **어긴** 게 아니라 SOP가 **재량으로 비워둔 틈**에 빠진 것 → 실행이 아니라 **교리** 격차.

### 설계 결정 3 — 2계층 그래프 (정적 교리 / 동적 오버레이)
- **Layer A (DoctrineGraph)** — SOP에서 추출한 정적 교리 지도. 노드+엣지. ①의 산출물.
- **Layer B (DebriefOverlay)** — 복기가 Layer A 노드 id에 부착되는 동적 학습층. ②의 산출물.

분리 이유: 교리 지도는 **한 번 만들어 캐시**(비싸고 안정적), 복기는 그 위에서 **라이브로 누적**. 같은 노드에 Lesson이 쌓이면서 재발이 드러난다.

### 설계 결정 4 — 격차 2분류로 고리를 닫음
모든 Lesson은 `gap.type` 으로 갈린다:

| 격차 | 무엇의 문제 | 신호 | 출력 |
|---|---|---|---|
| **doctrine** (교리) | *아는 것* | 대상 노드 modality가 약함(재량)·조항 부재·모호 | `sop_amendment` — 교리 편집안 |
| **execution** (실행) | *몸이 따르는 것* | 조항은 충분(의무/권고)한데 미준수 | `retraining_drill` — 재훈련 |

판정 원칙: **대상 노드의 modality를 먼저 본다.** mandatory/recommended인데 위반 → 실행 격차. discretionary/부재인데 안전 직결 → 교리 격차.

## 분류 체계 (Controlled Vocabularies)

### 분야 (`domain`) — 출처 5분야 + SSG
| 값 | 분야 | 번호대 |
|---|---|---|
| `command` | Ⅰ 지휘통제절차 | SOP 1xx |
| `fire` | Ⅱ 화재유형별 | SOP 2xx |
| `accident` | Ⅲ 사고유형별 | SOP 3xx |
| `ems` | Ⅳ 구급 단계별 | SOP 4xx |
| `situation` | Ⅴ 상황 단계별 | SOP 5xx |
| `safety` | Ⅵ 현장안전관리 표준지침 | SSG |

### 노드 타입 추출 단서 (`type`) — UI NodeChip 4타입과 1:1
| 타입 | 의미 | 원문 언어 단서 |
|---|---|---|
| `procedure` 절차 | 수행할 행동 단계 | 행위 동사·순차 단계 ("…정보를 수집한다", "…연결한다") |
| `decision` 판단점 | 판단·선택·분기 | "판단한다", "선택한다", "고려한다", "우선순위", "…한 경우" |
| `hazard` 위험 | 안전 임계 | 대원안전·긴급탈출·붕괴·고립·낙하물·유해 키워드 |
| `standard` 표준 | 규범적 의무·원칙·정의 | "…해야 한다", "책임을 진다", "원칙으로 한다", "우선순위는 다음과 같다" |

### Modality 추출 단서 (`modality`)
| 값 | 강제력 | 한국어 마커 |
|---|---|---|
| `mandatory` | 의무 | "…해야 한다", "…한다"(단정 지시), "책임을 진다", "삼가야 한다", "금지" |
| `recommended` | 권고 | "…을 원칙으로 한다", "…하도록 한다", "노력한다", "권장한다" |
| `discretionary` | 재량 | "…할 수 있다", "…을 고려한다", "필요시", "…수 있다" |
| `informational` | 정보 | "예)", 정의·배경 설명 ("…라 한다") |

### 엣지 관계 (`relation`)
| 값 | 의미 | 생성 방식 |
|---|---|---|
| `contains` | 계층 포함 (1→1.2→1.2.3) | **결정론적** (십진 번호), confidence=1 |
| `sequence` | 선후 (절차 순서) | 의미 추론 |
| `dependency` | 의존 (A 성립 후 B) | 의미 추론 |
| `exception` | 예외 ("다만/단" 단서) | 의미 추론 |
| `reference` | 교차 SOP 참조 | 의미 추론 |

## 파이프라인에서의 사용

**① 교리 지도화** → `DoctrineGraph` 생성:
1. SOP 한 섹션 텍스트 + 이 스키마를 Claude에 전달, 구조화 출력(tool use) 강제.
2. 노드 id는 원문 십진 번호로 결정론적 생성, `provenance.quote` 로 충실성 자가검증.
3. 결과를 `app/data/doctrine-map.json` 에 캐시.

**② 복기 반영** → `DebriefOverlay` 생성 (Layer A 위에서 라이브):
1. **Grounding** — 복기 문장 → 노드 id + confidence (`GroundingLink`).
2. **재발** — node_id 로 과거 Lesson 조회 → `recurrence`.
3. **격차 분류** — 대상 노드 modality 참조 → `gap.type`.
4. **생성** — doctrine→`sop_amendment`, execution→`retraining_drill`(`reference/8-step-training-model.md` 틀).

검증: 모든 산출물은 `doctrine-graph.schema.json` 통과가 게이트. 스키마 불일치 = Claude 재시도.
